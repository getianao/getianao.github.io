死锁-银行家算法

包括了安全性算法和资源请求算法。

例子：

假设现有3个进程P~1~-P~3~以及它们需要的最大资源和已拥有资源如下：

Allocation（已拥有资源）（A,B,C） max（需要的最大资源）
P1 1，2，4 2，5，8
P2 0，3，3 4，4，4
P3 4，1，1 5，4，4

系统中的资源如下

work（空闲资源）（A,B,C）
1，3，3

①检查安全性：

首先寻找一个进程满足need<work（需要资源小于空闲资源，说明这个进程可以满足），然后将work+allocation（空闲资源和已拥有资源相加）作为下一步的空闲资源，继续寻找可以被满足的进程；直至所有进程都可以被满足，说明现在的状态是安全的。

work（空闲资源）（A,B,C） Allocation（已拥有资源） need（需要的资源） work+allocation（完成后将被释放的资源） Finish（是否能完成）
P3 1，3，3 4，1，1 1，3，3 5，4，4 true
P1 5，4，4 1，2，4 1，3，4 6，6，8 true
P2 6，6，8 0，3，3 4，1，1 6，9，11 true

②检查资源请求是否能被满足

假如现在有一个资源请求是RequestP3（0，0，1），有work>RequestP3，可以满足这一步，假设执行了这一步将产生新的状态，再对新的状态做安全性检查，如果通过则说明这一资源请求可以被执行。

资源调度-理发师问题

理发店有一个理发师，他一次只能给一个人理发，理发店的休息区有若干椅子，当理发师在工作时，其他来店的客人只能依次在休息区的椅子上休息，等待理发师当前工作完成后再理发（事实上，即使理发师没有正在工作，顾客也是先进入休息区，在前往理发师处理发）；如果休息区的椅子坐满了，那没位置坐的顾客只能离开；如果理发店里没有顾客，则理发师在自己的位置上睡觉。

定义了三个信号量：

  • customers 表示顾客数量，如 customers 为 0 表示没有顾客。
  • barbers 表示理发师状态，如 barbers 为 0 表示理发师正在工作，为 1 表示理发师空闲。
  • mutex 是一个互斥量，用于保证空闲椅子数量的修改。
