摘录2 Understand the Activity Lifecycle

The Activity class provides a number of callbacks that allow the activity to know that a state has changed: that the system is creating, stopping, or resuming an activity, or destroying the process in which the activity resides.

有关activity状态的回调函数可以用来判断activtiy的状态，在合适的时候做出合适的反映。

Lifecycle callbacks

  • onCreate()

    In the onCreate() method, you perform basic application startup logic that should happen only once for the entire life of the activity.

    在onCreate()方法里我们做一些应用的基础逻辑设置，这只会在这个activity的生命周期中发生一次。

    如果有一个与这个activity关联了的 lifecycle-aware component（生命周期感知组件），它将会收到一个 ON_CREATE 事件，@OnLifecycleEvent 所注解的方法将会执行。

    As an alternative to defining the XML file and passing it to setContentView(), you can create new View objects in your activity code and build a view hierarchy by inserting new Views into aViewGroup. You then use that layout by passing the root ViewGroup to setContentView().

    当向 setContentView()传递一个xml布局资源文件的同时，创建了一个View对象并将它插入到Viewgroup中，然后将这个Viewgroup传给setContentView()。

  • onStart()

    The onStart() call makes the activity visible to the user, as the app prepares for the activity to enter the foreground and become interactive.

    onStart() 的时候activity将对用户可见，即将app放到前台并使之可以交互。

    When the activity moves to the started state, any lifecycle-aware component tied to the activity’s lifecycle will receive the ON_START event.

    当activity进入start状态，所有生命周期感知组件将会收到ON_START事件。

  • onResume()

    This is the state in which the app interacts with the user. The app stays in this state until something happens to take focus away from the app.

    在这个状态（resumed）下，app可以和用户交互，并且将一直保持这个状态直到某个事件发生使焦点从这个app转移，进入Paused状态。

    If you want the camera active only when the app is Resumed (visible and active in the foreground), then initialize the camera after the ON_RESUME event demonstrated above. If you want to keep the camera active while the activity is Paused but visible (e.g. in multi-window mode) then you should instead initialize the camera after the ON_START event.

    如果你想让你的app只在resumed状态下可用，你需要在ON_RESUME事件之后对相机进行初始化（并在ON_PAUSE释放）。如果你想让它在paused状态并可见下保持可用，你需要在ON_START状态下进行初始化（并在ON_STOP释放）。

    📌dialog不会触发pause,必须要是另外activtiy。在多窗口模式下，焦点的转换也会pause。

    Regardless of which build-up event you choose to perform an initialization operation in, make sure to use the corresponding lifecycle event to release the resource. If you initialize something after the ON_START event, release or terminate it after the ON_STOP event. If you initialize after the ON_RESUME event, release after the ON_PAUSE event.

    确保你的初始化和释放资源操作是在合适的生命周期下进行的。如果你在ON_START事件后初始化，也要在ON_STOP事件后释放。如果你在ON_RESUME事件后初始化，也要在ON_PAUSE事件后释放。

    Note, the code snippet above places camera initialization code in a lifecycle aware component. You can instead put this code directly into the activity lifecycle callbacks such as onStart() andonStop() but this is not recommended. Adding this logic into an independent, lifecycle-aware component allows you to reuse the component across multiple activities without having to duplicate code.

    除了将初始化卸载生命周期感知组件中，也可以直接写在生命周期的回调，但是不建议这么做因为生命周期感知组件可以帮助你在多个activity中复用组件。

  • onPause()

    There are several reasons why an activity may enter this state. For example:

    • Some event interrupts app execution, as described in the onResume() section. This is the most common case.
    • In Android 7.0 (API level 24) or higher, multiple apps run in multi-window mode. Because only one of the apps (windows) has focus at any time, the system pauses all of the other apps.
    • A new, semi-transparent activity (such as a dialog) opens. As long as the activity is still partially visible but not in focus, it remains paused

    activity进入PAUSED阶段的情况：

    ①一些中断事件

    ②多窗口模式切换焦点

    ③半透明activity

    ❓中断事件是什么也没有说

    onPause() execution is very brief, and does not necessarily afford enough time to perform save operations. For this reason, you should not use onPause() to save application or user data, make network calls, or execute database transactions; such work may not complete before the method completes. Instead, you should perform heavy-load shutdown operations during onStop()

    onPause()操作使非常简短的，它不能提供足够的时间去进行保存操作，所以不要在onPause()中进行保存操作、数据操作、网络操作或数据库操作；你应当将这些负担较重的结束操作移至 onStop()。

  • onStop()

    When your activity enters the Stopped state, the Activity object is kept resident in memory: It maintains all state and member information, but is not attached to the window manager. When the activity resumes, the activity recalls this information. You don’t need to re-initialize components that were created during any of the callback methods leading up to the Resumed state. The system also keeps track of the current state for each View object in the layout, so if the user entered text into an EditText widget, that content is retained so you don’t need to save and restore it.

    当activity处于stop状态时，内存中将会保留activtiy的实例，包括所有状态和成员信息，但是它没有与窗口管理器相连接。系统也保留了布局中所有view的状态 ，比如输入框中的信息。

    ❓不应该是“leading up to the started state”吗？

    Note: Once your activity is stopped, the system might destroy the process that contains the activity if the system needs to recover memory. Even if the system destroys the process while the activity is stopped, the system still retains the state of the View objects (such as text in an EditText widget) in a Bundle (a blob of key-value pairs) and restores them if the user navigates back to the activity.

    当一个activity处于stopped状态时，系统可能因为内存不足而销毁含activity的进程。即使系统中的怎么做了，系统还是在Bundle中储存了信息（手动）。

  • onDestroy()

    onDestroy() is called before the activity is destroyed. The system invokes this callback either because:

    1. the activity is finishing (due to the user completely dismissing the activity or due tofinish() being called on the activity), or
    2. the system is temporarily destroying the activity due to a configuration change (such as device rotation or multi-window mode)

    以下情况下会调用onDestroy()

    ①当activity被finish（用户退出或finish()被调用）

    ②配置改变导致系统暂时destroy了activity（比如多窗口程序或旋转屏幕）

    Instead of putting logic in your Activity to determine why it is being destroyed you should use a ViewModel object to contain the relevant view data for your Activity. If the Activity is going to be recreated due to a configuration change the ViewModel does not have to do anything since it will be preserved and given to the next Activity instance. If the Activity is not going to be recreated then the ViewModel will have the onCleared() method called where it can clean up any data it needs to before being destroyed.

    不用在onDestroy中加入逻辑考虑是什么导致了Destroy，只需要将相关的view数据集合在ViewModel中，如果是因为配置改变，则ViewModel什么也不用做，因为它将被传递到新的Activity中；否则它的onCleared()方法会被调用用来清除所有数据。

Activity state and ejection（抛出） from memory

The system kills processes when it needs to free up RAM; the likelihood of the system killing a given process depends on the state of the process at the time. Process state, in turn, depends on the state of the activity running in the process. Table 1 shows the correlation among process state, activity state, and likelihood of the system’s killing the process.

Likelihood of being killed Process state Activity state
Least Foreground (having or about to get focus) Created Started Resumed
More Background (lost focus) Paused
Most Background (not visible) Stopped
Empty Destroyed

系统杀死一个进程取决于进程的状态，进程的状态取决于进程里运行的activity的状态；

The system never kills an activity directly to free up memory. Instead, it kills the process in which the activity runs, destroying not only the activity but everything else running in the process, as well.

系统因为activity的状态杀死了一个进程，意味着这个进程中的所有activity都会被杀掉。

Saving and restoring transient UI state

使用情形：当你的配置发生改变或切换到另一个app中在切换回来，你希望UI的状态不要发生变化，但是可能activtiy已经被destroy了。

如果时轻量简单的数据保存只用 onSaveInstanceState()就行，复杂的需要 ViewModel, onSaveInstanceState()配合使用。

  • Instance state

    假如是因为系统约束而不是主动（back or finish）去destroy，activity实例实际上已经不在了，但系统认为它还在，当返回这个activity时，系统会利用保存的数据来重新创建一个activity，这样的系统用于恢复之前状态的数据被称为Instance state，他是一个保存在Bundle对象中键值对的集合。你不需要去操作这些数据使它会到之前的状态，但是你需要保存足够多的状态信息。

  • Save simple, lightweight UI state using onSaveInstanceState()

    Note: onSaveInstanceState() is not called when the user explicitly closes the activity or in other cases when finish()is called.

    onSaveInstanceState()不能被直接调用。

    To save persistent data, such as user preferences or data for a database, you should take appropriate opportunities when your activity is in the foreground. If no such opportunity arises, you should save such data during the onStop() method.

    保存持久化的数据，你需要找到合适的机会当你的程序在前台时执行，如果没有合适的机会，可以在onStop() 方法中执行。

  • Restore activity UI state using saved instance state

    Instead of restoring the state during onCreate()) you may choose to implementonRestoreInstanceState(), which the system calls after the onStart() method. The system calls onRestoreInstanceState() only if there is a saved state to restore, so you do not need to check whether the Bundle is null:

    除了在onCreate方法中恢复数据，也可以在onRestoreInstanceState()中恢复，不过它不用判断是否存在保存的状态。

    Caution: Always call the superclass implementation of onRestoreInstanceState() so the default implementation can restore the state of the view hierarchy.

    一定要在onRestoreInstanceState()调用子类的onRestoreInstanceState()，这样才能层次的恢复view的状态。

Navigating between activities

  • Coordinating activities

    The order of lifecycle callbacks is well defined, particularly when the two activities are in the same process (app) and one is starting the other. Here’s the order of operations that occur when Activity A starts Activity B:

    1. Activity A’s onPause() method executes.

    2. Activity B’s onCreate(), onStart(), and onResume() methods execute in sequence. (Activity B now has user focus.)

    3. Then, if Activity A is no longer visible on screen, its onStop() method executes.

    从activity A启动另一个Activity B，我们认为的生命周期变化是：A‘s onPause()-(如果A不可见了)->A’s onStop() ->B‘s onCreate() ->B‘s onStart() ->B‘s onResume()

    但其实是这样的：A‘s onPause() ->B‘s onCreate() ->B‘s onStart() ->B‘s onResume() -(如果A不可见了)->A’s onStop()
